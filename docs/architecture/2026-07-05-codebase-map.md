## 1. Executive summary

orchestrai is a personal AI-team-orchestrator plugin for Claude Code, applied
across the owner's own repos. It contains no application runtime. The whole
system is built from four kinds of material: agent persona definitions
(markdown with YAML frontmatter), skill procedure documents (SKILL.md files the
model follows step by step), three JavaScript orchestration scripts executed by
Claude Code's Workflow tool, and a documentation tree that records decisions,
plans, reviews, and research. A zero-dependency test suite (Node's built-in
test runner, no packages) locks in the cross-cutting policies the scripts and
agent files must not regress.

The design center is a flat-star team model. One lead session (the main Claude
Code session, running a tm- skill) dispatches five role agents, one bounded job
each: architect (sub-plans, splits, arbitration), developer (one issue end to
end in a worktree), tester (independent verification), reviewer (spec pass then
quality pass), and fact-checker (claim auditing). Agents never call each other.
Each returns a fixed-shape text report to the lead, and durable state lives in
GitHub (issue comments, draft PRs, labels), so a dropped session resumes from
GitHub rather than from memory.

Three layers sit on top of that team. The tm- skills are the user-facing
command surface: tm-grill-me and tm-to-issues turn a raw need into sized
issues, tm-kickoff fans issues out through a serial
architect-developer-tester-reviewer pipeline (up to 3 packages concurrently),
and tm-advisor wraps the same pipeline in a batch operating model with one
human sign-off per batch. The three workflow scripts (tm-review-changes,
tm-review-codebase, tm-map-codebase) are token-bounded by construction: agent
counts are fixed or capped in the script, and every stage pins its own model
and effort (Sonnet workers, one Fable critic), so no run can inherit the
session model or fan out unboundedly. The plugin manifest and shared config
layer makes all of this installable: a marketplace descriptor points at
.claude/ as the plugin source, settings.json enables the superpowers plugin,
and team-guide.md, imported into every session via CLAUDE.md, is the process
and model-policy rulebook the rest of the machinery is written against.

The docs tree closes the loop. docs/architecture/ holds the locked identity
decision (operating-model.md) and generated dated maps like this one.
docs/superpowers/specs/ and docs/plans/ hold the spec-then-plan paper trail
behind each non-trivial build, with supersession banners and dated amendments
layered on rather than rewrites. docs/reviews/ and docs/research/ hold the
generated review reports, the orchestration comparison that justified the
model policy, and third-party research, with docs/team-architecture.md as the
standing authored reference they all cite.

## 3. Data-flow narrative

Session load. When a Claude Code session opens in this repo, the harness reads
CLAUDE.md, whose `@.claude/team-guide.md` import inlines the team guide into
context. settings.json is read at the same point to enable the superpowers
plugin. For consumers who install rather than clone, marketplace.json is read
once at marketplace-add time to locate the plugin source (./.claude), and
plugin.json identifies and versions the install. This puts the process rules,
model policy, and label vocabulary in scope before any command runs.

Refinement to issues. A raw need enters as conversation text. tm-grill-me
resolves it through a one-question-at-a-time interview; tm-to-issues slices
the approved plan into sized GitHub issues via `gh issue create`, with
dependencies as literal `Blocked by: #N` lines. For substantive builds the
same input first produces a dated design spec under docs/superpowers/specs/
and then an implementation plan under docs/plans/, which reference each other
by path.

Dispatch through the team. tm-kickoff reads issue bodies, comments, labels,
and open-PR state via gh, builds a dependency-ordered wave plan, and runs each
package through the serial pipeline. Each agent receives a small explicit
payload (an issue number, a branch+issue pair, a PR+issue pair, a job tag, or
a quoted text block) and re-derives further context itself from GitHub and git
rather than receiving a transcript. Results flow back two ways: a fixed-shape
text report to the lead (STATUS/VERDICT-first templates the lead parses), and
durable GitHub writes (sub-plan comments, draft PRs, verdict comments, labels)
that a later session can resume from. developer, tester, and fact-checker each
operate in an isolated git worktree with a detached-HEAD checkout; tm-kickoff
performs deterministic worktree cleanup at wave end. tm-advisor wraps this
same flow in a batch: package issues plus a `Batch: <slug>` tracking issue
carry the approved contract, the decision log, parked questions, and the final
report, and the resume procedure re-enters from that issue's state.

Bounded workflows. The three thin-wrapper skills pass at most one argument (a
base ref, a path, an area ceiling) to the matching .claude/workflows/*.js
script via the Workflow tool and relay its output verbatim. Inside a
scout-based script, args flow through parseArgs and safeRef, the scout returns
a MAP_SCHEMA area partition, the script clamps it to MAX_AREAS in plain JS,
one prompt per area fans out through parallel(), each worker returns a typed
schema object (null-padded on failure), and the accumulated outputs plus
coverage bookkeeping are serialized into one critic prompt. The critic writes
the dated markdown file (docs/reviews/<date>-codebase-review.md or
docs/architecture/<date>-codebase-map.md) and returns the REPORT_SCHEMA
object, which becomes the script's return value and the skill's verbatim
report. tm-review-changes skips the scout and fans out over a fixed dimension
list against `git diff <base>...HEAD`.

Documentation and record. Generated reports land as new dated files; older
ones are never edited, only marked superseded with forward pointers. Later
reviews read earlier reviews to state what shipped and what stayed open.
Specs and plans absorb changes as dated amendments or top-of-file supersession
banners. operating-model.md points outward to the files that own each
mechanic; the codebase map flows the other way, produced from the whole repo
as a write target. When code and a doc disagree, the code wins and the doc is
corrected in the same PR.

Tests. `npm test` (package.json) runs the __tests__/*.test.mjs files, which
read the workflow sources, agent frontmatter, and the tm-new-project CI
template as text: effort-policy asserts every stage's model/effort pin,
helpers executes the real parseArgs/safeRef bodies via node:vm and asserts
byte-identity of the duplicated copies, and ci-template-policy asserts the CI
template still encodes the cost policy. The tests consume source text only and
play no part in the scripts' runtime flow.

## 4. Dependency summary

External dependencies (no npm packages anywhere; package.json is private with
no dependency fields):

- Claude Code harness: CLAUDE.md loading and @-imports, the plugin/marketplace
  system (reads marketplace.json and plugin.json), settings.json, the
  subagent/Task machinery that runs the role agents, and the Workflow tool
  runtime that supplies the ambient `agent`, `parallel`, `phase`, and `args`
  bindings the three scripts reference but never define.
- GitHub via the `gh` CLI: issues, PRs, labels, comments, and search are both
  the input source and the durable state store for tm-kickoff, tm-advisor,
  tm-to-issues, tm-new-project, and every role agent.
- git: branch, diff, log, worktree, detached-HEAD checkout, and
  force-with-lease push mechanics across developer/tester/fact-checker and the
  workflow scripts' shelled-out commands.
- Node.js built-ins only: node --test, node:assert/strict, node:vm, node:fs,
  node:url, node:path, used exclusively by the test suite.
- superpowers@claude-plugins-official plugin: enabled in settings.json;
  developer preloads test-driven-development, tester preloads
  verification-before-completion, and the plans area names brainstorming,
  writing-plans, and executing-plans as its authoring tools.
- Upstream community sources: mattpocock/skills (MIT) as the adapted source of
  tm-grill-me and tm-to-issues (recorded in THIRD_PARTY_NOTICES.md);
  vijaythecoder/awesome-claude-agents and owainlewis/youtube-tutorials as
  referenced-only research subjects.
- GitHub Actions, indirectly: templates/ci.yml is a reference workflow copied
  into new repos, not executed here.

Cross-area dependencies:

- team-guide.md (shared config) is the policy source the agent frontmatter,
  workflow pins, and skill label blocks implement; effort-policy.test.mjs
  (workflows area) reads the agent files (agents area) to enforce it.
- The tm- skills dispatch the role agents (tm-kickoff, and tm-advisor through
  it) and invoke the workflow scripts (the three thin wrappers), and consume
  the agents' report contracts.
- tm-advisor implements the advisor design spec (specs area) and the
  operating-model doc's intake stage; operating-model.md in turn points at the
  skills, team-guide.md, agents, workflows, and docs/team-architecture.md as
  the owners of its loop stages.
- The workflow scripts write into docs/reviews/ and docs/architecture/; the
  reviews area's dated reports are their output, and newer reports read older
  ones. ci-template-policy.test.mjs (workflows area) reads
  tm-new-project/templates/ci.yml (skills area).
- Deliberate byte-identical duplication substitutes for imports in two places:
  parseArgs/safeRef and the schema fragments across the three workflow scripts
  (the Workflow runtime has no shared imports; helpers.test.mjs enforces
  identity), and the six-label `gh label create --force` block across
  tm-advisor, tm-kickoff, and tm-new-project. Branch/worktree checkout
  mechanics repeat near-verbatim across developer, tester, and fact-checker.
- Root docs anchor the rest: CLAUDE.md imports team-guide.md and names the
  docs locations; README.md enumerates the agents, skills, and workflows and
  points at THIRD_PARTY_NOTICES.md; package.json's test script points at the
  workflows' __tests__/.

## 5. Open questions

- The Workflow runtime contract is assumed, not defined: `agent`, `parallel`,
  `phase`, and `args` are ambient globals the three scripts call but nothing
  in the repo declares. Their exact semantics (failure handling, null-padding,
  schema enforcement) can only be confirmed against the Claude Code host, not
  from source here.
- operating-model.md defers the physical removal of NEW-PROJECT-SETUP.md and
  .claude/skills/tm-new-project/ to a later usability batch. The timing is a
  human decision, and the fate of templates/ci.yml, which
  ci-template-policy.test.mjs reads, once that scaffolding is retired is
  undetermined.
- The measured ultracode trial's agent count appears as 288 in
  team-guide.md's model policy and as 297 in the area description of
  docs/reviews/2026-06-30-orchestration-comparison.md. Which figure the
  report actually states needs a human check.
- team-guide.md declares settings.json split by ownership (enabledPlugins
  template-managed, permissions/hooks/env/defaultMode project-owned), but
  whether `/plugin update` actually preserves the project-owned keys is host
  behavior not verifiable from this repo.
- Whether the marketplace.json `renames` entry (sv-tmueller to orchestrai)
  migrates existing installs under the old name automatically is Claude Code
  host behavior the manifest alone does not determine.
